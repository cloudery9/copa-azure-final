## Story

**Como** integrante do squad de desenvolvimento do Living Lab (papel `@dev`, com o desenho de teste de `@qa`, sob o gate `@qa`),
**eu quero** adicionar um teste de integração real para a idempotência do INSERT em `purchases` (hoje só validada manualmente/e2e), testes unitários para o `gatewayTrust.js` (hoje com **zero** testes no backend Node), e testes para os guards fail-closed de startup (config ausente → `InvalidOperationException`, hoje sem cobertura em nenhum dos 3 serviços .NET),
**para que** uma regressão futura nesses 3 mecanismos críticos (idempotência de compra, confiança do gateway, config fail-closed) seja pega por CI, não descoberta em produção — fechando a lacuna que a ADE-009 explicitamente aponta: *"guards de startup sem teste (regressão do `common` passaria hoje)"* e *"backend Node = 0 testes"*.

### Bloco B — Integração: idempotência SQL real (ADE-009 §Consequences item 2 / plano §5 item 2)

2. **AC-2 — Novo teste de integração cobre `PurchaseRepository.InsertPurchaseAsync` contra um SQL Server REAL** (não mockado): usa **Testcontainers.MsSql** (pacote NuGet — confirmado ausente hoje em qualquer `.csproj` do repo, `grep Testcontainers` = 0 ocorrências) para subir um container efêmero do SQL Server, aplicar o schema real (`fifa2026-api/database/schema.sql` — reúso, não recriação de DDL) e as migrations necessárias (`ticket_categories`/`matches`/`teams` mínimos para o teste), e confirmar que **duas** chamadas com o **mesmo** `CorrelationId` retornam `InsertOutcome.Inserted` na primeira e `InsertOutcome.Duplicate` na segunda — via a violação REAL do índice único (`SqlException` 2627/2601, verificado `PurchaseRepository.cs` linhas 111-119), não uma exceção simulada em mock. Hoje essa idempotência só é validada **manualmente/e2e** (ADE-009 §Consequences: "teste de integração da idempotência SQL real, hoje só e2e manual").
   *(Fonte: verificado `PurchaseRepository.cs` linhas 17-20 (`SqlUniqueViolation = 2627`, `SqlDuplicateKey = 2601`), 111-119 (catch); grep `Testcontainers` em `src/` = 0 ocorrências; ADE-009 §Consequences item de Consolidação.)*

3. **AC-3 — O teste cobre também o caso de concorrência (2 requests simultâneas, mesmo `CorrelationId`):** ambas disparadas em paralelo (`Task.WhenAll`) contra o mesmo container — exatamente **uma** retorna `Inserted`, a outra `Duplicate` (nunca as duas `Inserted`, nunca exceção não tratada) — prova que a defesa é a violação do índice único no banco (ADE-000 Inv 4: "NUNCA SELECT-then-INSERT, evita TOCTOU race"), não uma checagem prévia frágil.
   *(Fonte: verificado comentário `PurchaseRepository.cs` linha 13 — "NUNCA SELECT-then-INSERT, evita TOCTOU race"; ADE-000 Inv 4.)*

4. **AC-4 — Graceful degradation se Docker indisponível:** o teste de integração é anotado/categorizado de forma que ambientes sem Docker (ex.: alguns runners locais) possam pulá-lo explicitamente (ex.: `[Trait("Category","Integration")]` + filtro no `dotnet test`, ou skip condicional via `DockerAvailableFact`-like helper) — não quebra o build geral; CI (GitHub Actions `ubuntu-latest`, que já tem Docker) roda o teste normalmente.
   *(Fonte: boa prática de testes de integração com Testcontainers — direção, não código; nenhum mecanismo de skip específico é inventado além do já padrão em xUnit.)*

### Bloco C — Unit tests do `gatewayTrust.js` (ADE-009 §Consequences item 4 / plano §5 item 4)

5. **AC-5 — Framework de teste JS adicionado ao backend v1:** `fifa2026-api/package.json` **não tem nenhuma dependência de teste hoje** (verificado — sem `jest`/`mocha`/`vitest` em `devDependencies`; `scripts` só tem `start`/`dev`). Esta AC adiciona **Jest** (escolha natural para o ecossistema Node/Express já em uso — `express`, `jsonwebtoken`, `bcryptjs`) como `devDependency` + um script `"test": "jest"` em `package.json`.
   *(Fonte: verificado `fifa2026-api/package.json` completo — 0 dependências de teste; glob `fifa2026-api/**/*.test.js` e `*.spec.js` e `__tests__/**` = 0 arquivos.)*

6. **AC-6 — Testes unitários de `gatewayTrust.js` cobrem os cenários que hoje têm zero cobertura automatizada:**
    - `safeEqual(a, b)`: retorna `true` só quando ambos são strings de mesmo tamanho e conteúdo idêntico; retorna `false` (nunca lança) para tipos não-string, tamanhos diferentes, ou valores divergentes (verificado `gatewayTrust.js` linhas 28-38).
    - `gatewayTrustMiddleware`: com `GATEWAY_SHARED_SECRET` configurado e `X-Gateway-Key` correto → `req.user` populado (`role: 'admin', source: 'gateway'`) e `next()` chamado (linhas 49-56).
    - Com `GATEWAY_SHARED_SECRET` **vazio/ausente** → cai no fluxo legado (`authMiddleware` → `adminMiddleware`), **independente** do header recebido (linha 60) — prova o comportamento fail-open-para-legado documentado nas linhas 17-18.
    - Com secret configurado mas header **ausente ou incorreto** → cai no fluxo legado (não autentica como admin via gateway) — nunca "meio-autenticado".
    *(Fonte: verificado `gatewayTrust.js` linhas 1-64 completo — arquivo lido integralmente nesta story.)*

7. **AC-7 — Teste NÃO precisa provar tempo-constante empiricamente (limite reconhecido):** medir timing em microbenchmark de unit test é frágil/não-determinístico; o teste confirma o **comportamento funcional** de `safeEqual` (retorna `false` seguramente para tamanhos diferentes, sem lançar) — a garantia de tempo-constante em si vem de usar `crypto.timingSafeEqual` (API nativa do Node, já correta no código, linha 37) e não é reprovada por unit test. Documentado como limitação conhecida, não fingida como coberta.
    *(Fonte: honestidade de escopo, Art. IV — não inventar uma garantia de teste que não é praticável.)*

### Bloco D — Guards fail-closed de startup (ADE-009 §Consequences item 6 / plano §5 item 6)

8. **AC-8 — Guards de configuração ausente no Gateway testados (hoje sem cobertura):** `Gateway/Program.cs` lança `InvalidOperationException` em 4 pontos verificados (linhas 245-252, 254-260, 278-285, 287-293) quando `Jwt:CiamTenantId`/`Jwt:CiamClientId`/`Jwt:AdminTenantId`/`Jwt:AdminClientId` estão ausentes ou o tenant é `"common"`. Novo(s) teste(s) em `Fifa2026.V2.Gateway.Tests` (reusando `GatewayTestFixture`/`WebApplicationFactory`, o mesmo padrão já usado por `JwtValidationTests.cs`/`DualIssuerTests.cs`) confirmam que **iniciar** o host com configuração faltando/`"common"` lança/falha a inicialização — regressão aqui hoje **passaria silenciosamente** (nenhum teste atual cobre esses 4 guards).
   *(Fonte: verificado `Gateway/Program.cs` linhas 245-293 — 4 blocos de guard; listagem de `Fifa2026.V2.Gateway.Tests/` confirma ausência de arquivo dedicado a esses guards.)*

9. **AC-9 — Guards de `SqlConnectionString` ausente testados em `PurchaseRepository`/`FifaQueryRepository` (Functions/McpServer):** ambos os construtores lançam `InvalidOperationException` quando `configuration["SqlConnectionString"]` é nulo/vazio (verificado `PurchaseRepository.cs` linhas 25-30, `FifaQueryRepository.cs` linhas 22-28). Novo teste unitário direto (instancia a classe com um `IConfiguration` sem a chave, sem precisar de `WebApplicationFactory`) confirma o `throw`. Hoje **nenhum teste existente** cobre esse caminho (confirmado: nenhum arquivo em `Fifa2026.V2.Functions.Tests`/`Fifa2026.V2.McpServer.Tests` testa o construtor com config ausente).
   *(Fonte: verificado `PurchaseRepository.cs` linhas 22-31; `FifaQueryRepository.cs` linhas 17-28.)*
